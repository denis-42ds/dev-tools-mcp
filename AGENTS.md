## Роли и ответственности

### MCP Host (Claude)
- Принимает запросы пользователя на естественном языке
- Определяет, какой инструмент использовать и с какими параметрами
- Интерпретирует структурированный вывод инструментов
- Формулирует финальный ответ пользователю
- Может вызывать несколько инструментов последовательно (например: scan_tech_debt → генерация задач для Jira)

### MCP Server (dev-tools-mcp)
Детерминированный исполнитель без состояния. Получает параметры → возвращает Markdown-отчёт.

#### Tool 1: `scan_tech_debt`
**Входные данные:** путь, фильтры  
**Алгоритм:**
1. Рекурсивный обход файловой системы (исключая .git, node_modules и т.д.)
2. Для каждого файла — чтение построчно
3. Regex-поиск маркеров (TODO/FIXME/HACK/BUG/XXX/DEPRECATED/NOTE/OPTIMIZE)
4. Классификация по приоритету
5. Сортировка: critical → warning → low, затем по файлу/строке
6. Формирование Markdown-отчёта со статистикой и рекомендациями

**Выходные данные:** Структурированный Markdown с иерархией приоритетов

#### Tool 2: `generate_release_notes`
**Входные данные:** путь к репо, from_ref, to_ref, флаги  
**Алгоритм:**
1. Валидация git-репозитория (git rev-parse --git-dir)
2. Автоопределение диапазона (если from_ref не задан — берём предпоследний тег)
3. Получение git log с форматом `hash|author|date|subject|body`
4. Парсинг Conventional Commits regex: `^(type)(\(scope\))?(!)?:\s(desc)`
5. Категоризация по типам (feat/fix/perf/refactor/docs/test/build/ci/chore)
6. Выделение BREAKING CHANGES (! или "BREAKING CHANGE" в теле)
7. Сбор статистики авторов
8. Формирование Markdown в GitHub Releases формате

**Выходные данные:** Готовые Release Notes в Markdown

#### Tool 3: `audit_dockerfile`
**Входные данные:** путь (файл или директория), strict-флаг  
**Алгоритм:**
1. Поиск всех Dockerfile/Dockerfile.* рекурсивно
2. Для каждого файла — парсинг инструкций (FROM/RUN/COPY/ENV/USER/etc.)
3. Применение 15 правил (3 critical + 8 warning + 4 info)
4. Подсчёт score: 100 - critical×20 - warning×5 - info×1
5. Формирование отчёта с рекомендациями по каждой проблеме

**Выходные данные:** Аудит-отчёт с оценкой по каждому Dockerfile

## Поток взаимодействия (пример)

```
Пользователь: "Подготовь release notes для v1.0.0 и найди весь критичный 
               техдолг, который нужно закрыть перед релизом"

Claude (MCP Host):
  1. generate_release_notes(repo_path=".", from_ref="v0.1.0", to_ref="v1.0.0")
     → Markdown с коммитами
  
  2. scan_tech_debt(path=".", priority_filter="high")
     → Список критичных маркеров
  
  3. Синтез: "Вот release notes. Перед публикацией рекомендую закрыть 
              5 критичных задач: [список из scan_tech_debt]"
```

## Принципы проектирования

### Детерминизм
Каждый инструмент при одинаковых входных данных возвращает одинаковый результат. Нет случайности, нет LLM внутри инструментов.

### Fail-fast с понятными ошибками
Если путь не существует, не является git-репозиторием или Dockerfile не найден — немедленно возвращаем читаемое сообщение об ошибке вместо исключения.

### Безопасность
- Сервер работает с правами непривилегированного пользователя
- Монтирование директорий только read-only
- Нет исполнения кода из анализируемых файлов
- Нет сетевых запросов

### Расширяемость
Добавить новый инструмент = написать функцию с декоратором `@mcp.tool()`.  
Добавить правило в аудит = добавить функцию `add(...)` в `_audit_single_dockerfile`.

## Процесс разработки с AI

### Этап 1: Определение проблемы
Использовался Claude для исследования типичных pain points разработчиков.  
Вывод: три наиболее частые ручные операции — поиск техдолга, написание release notes, аудит Dockerfile.

### Этап 2: Проектирование API
Claude помог сформулировать схему параметров инструментов:
- Минимальный обязательный параметр (path/repo_path)
- Необязательные фильтры с разумными дефолтами
- Вывод всегда в Markdown (читаемо и для человека, и для LLM)

### Этап 3: Генерация кода
- Claude генерировал реализацию каждого инструмента
- Ревью: проверка edge cases (пустой репозиторий, нет тегов, нет коммитов)
- Итерация: добавление поддержки binary files, skip-директорий, encoding errors

### Этап 4: Тестовый проект
Claude намеренно написал "плохой" код с реальными маркерами техдолга и "плохой" Dockerfile с реальными проблемами для достоверного демо.

### Что сработало хорошо
- Генерация regex для Conventional Commits — точная с первой попытки
- Документация инструментов (docstrings) — сгенерирована AI и сразу хорошего качества
- Структура проекта и Dockerfile для самого сервера

### Что потребовало итераций
- Логика автоопределения диапазона в generate_release_notes (edge case: нет тегов)
- Обработка git log с нестандартными символами в commit messages
- Правила аудита Dockerfile: потребовалось несколько итераций для снижения false positives
